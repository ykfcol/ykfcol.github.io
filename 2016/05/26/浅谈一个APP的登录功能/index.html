浅谈一个APP的登录功能

发表于 2016-05-26   |   分类于 Android   |     |   阅读次数

记得又一次跟同事出去吃饭，突然心血来潮问他，APP的登录功能需要注意什么问题？登录功能是一个APP的基本功能，当时他如数家珍的给我列出了一些，什么登录后要注意登录数据的安全（加密），最好加上验证码防止有人恶意刷库，注销的时候记得清除保存的登录信息等等，可能大部分人都会忽略这个功能的重要性，但我觉得登录是考察一个APP开发人员是否合格的标准。

登录信息的存储

最直接的做法就是，登录成功后本地保存用户名和密码，每次重启APP判断本地是否有保存用户名和密码，如果有，则将用户名和密码传入登录接口，模拟用户登录的行为。包括我在项目快速迭代的早期也是这样做的，现在想想，有点小庆幸。那么这样做会带来怎样的安全风险？

本地保存用户密码，这种敏感数据容易被人窃取。要么是在本地文件看到这些信息，要么是侦听App的网络请求，尝试获取请求数据。这就要求我们在本地保存用户密码时，对其进行加密，而且要不对称加密，对称加密容易被有心人反向推算出来。

那我们在登录之后呢？跟用户相关的数据是怎样请求的？部分APP可能会在本地保存一个isLogin的全局变量，登录成功后设置为true，然后在涉及用户相关的接口请求时，将用户的UserId传递过去，服务器那边检测是否有UserId，返回相应数据，这样做的安全风险是非常大的。有一种补救的办法，是在调用接口的时候将UserId和加密后的密码一起传过去，服务器通过加密后的密码比对来决定是否返回用户数据。

Token

有没有一种方式，可以不在本地保存用户名和密码就能实现用户相关的数据调用？其实Web在这点已经做得很成熟了，那就是Cookie机制。

Cookie，也可以说是Token，它是一种为了辨别用户身份而存储在本地的数据，通常是经过加密的。它是用户身份的唯一性标志。我们可以这样设计用户的登录：在用户登录成功后，会从服务器获取到一个Token，然后保存在本地，我们在每次请求用户相关数据的时候，把Token传给服务器，服务器去判断Token是否有效，有效则返回对应的用户数据。我们在判断用户是否登录的时候，也就可以通过判断Token是否为空来确定。

使用Token需要注意的几点：

  • Token过期，需要提示用户重新登录
  • 用户注销，清除Token
  • 请求成功后，覆盖本地Token

我们可以在基类里封装Token过期时的统一处理。

防止恶意刷库

对于用户而言，他们通常会在各个网站使用相同的用户名和密码，一个网站泄漏了，各个网站的信息也失去了保障。那我们要如何规避这种风险？这就要用到我上文提到的验证码了。

传统的做法就是给登录接口添加验证码参数。现在市面上大多APP都做的很好，用手机号进行注册，登录时发送手机验证码验证用户身份，那么新的问题又来了，要是有人刷短信呢（频繁发送短信，我们一般会做短信发送的时间控制）？这就需要我们在服务器做相应的处理了，服务器在发现同一IP短时间内频繁访问一个接口，就返回对应的状态码，去验证用户的身份。
